# Безопасность проекта «Кладовая»

## Что защищено

- приложение и PostgreSQL слушают только localhost; извне доступны только HTTPS и SSH;
- вход ограничен по частоте, сессия хранится в `HttpOnly`/`Secure`/`SameSite=Strict` cookie;
- изменяющие API-запросы проверяют источник, размер и формат данных;
- договоры, чеки и новые резервные копии шифруются AES-256-GCM до загрузки в приватный S3;
- загружаемые документы проверяются по сигнатуре, а не только по расширению;
- чек из письма не подтверждает оплату автоматически: сотрудник обязан сверить поступление с банком;
- production использует поддерживаемую Node.js LTS, изолированную systemd-службу, firewall и fail2ban;
- парольный SSH-вход запрещён, разрешены только ключи.

## Обязательные правила эксплуатации

1. Не передавать пароль администратора сотрудникам. Для рабочих пользователей нужны отдельные учётные записи и MFA.
2. Хранить `DOCUMENT_ENCRYPTION_KEY` вне сервера в корпоративном менеджере паролей. Без него зашифрованные документы и резервные копии восстановить нельзя.
3. Не считать изображение чека доказательством оплаты. Статус «оплачено» ставится после сверки банковского поступления.
4. Никогда не добавлять `.env`, SSH-ключи, пароли почты и S3-ключи в Git.
5. Не реже раза в месяц проверять восстановление резервной копии на отдельной тестовой базе.

## Оставшиеся задачи перед выдачей доступа сотрудникам

- реализовать персональные учётные записи, роли и журнал действий;
- подключить второй фактор входа;
- перенести старые резервные копии, созданные до включения прикладного шифрования, в новый зашифрованный формат;
- настроить централизованный мониторинг ошибок и уведомления о подозрительных входах.
