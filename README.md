# Кладовая

Внутренняя single-tenant система учета аренды кладовок, гаражей и боксов.

## Что реализовано

- вход и три роли: Admin, Manager, Accountant;
- dashboard с operational KPI и фильтром по объекту;
- реестры объектов, юнитов, клиентов, договоров, начислений, оплат, задач и пользователей;
- карточка клиента со связанными договорами, начислениями, оплатами, документами и задачами;
- создание основных сущностей в демо-режиме;
- проверка пересечения активных договоров;
- автоматический статус занятого юнита;
- статусы начислений pending / partial / paid / overdue;
- банковский QR `ST00012` по реквизитам с договором, месяцем и суммой;
- экран платёжных реквизитов и подготовка письма клиенту;
- журнал запросов на оплату как основа почтовой автоматизации;
- PostgreSQL-схема Prisma, SQL-миграция, seed и тесты доменной логики.

## Быстрый запуск

Требуется Node.js 20+.

```bash
pnpm install
pnpm dev
```

Откройте `http://localhost:3000`. Демо-данные сохраняются в Local Storage браузера. Пароль на экране входа демонстрационный; можно выбрать любую из трех ролей.

Опубликованная версия: `https://radiovanya.github.io/kladovaya/`.

## Production: PostgreSQL и закрытое S3

```bash
cp .env.example .env
pnpm db:generate
pnpm db:migrate
pnpm db:seed
```

Production-сборка работает через серверные route handlers Next.js:

- общие данные сотрудников сохраняются в PostgreSQL;
- вход защищён HTTP-only сессией;
- договоры загружаются в приватный S3-бакет и открываются по временным ссылкам;
- QR отправляется через SMTP после заполнения почтовых переменных окружения;
- ежедневный `pg_dump` запускается systemd-таймером и сохраняется в S3.

GitHub Pages остаётся демонстрационной статической версией и не содержит production-секретов.

## Архитектурные решения

- монолитный Next.js-клиент с отдельным чистым слоем бизнес-правил;
- PostgreSQL + Prisma как целевая персистентная модель;
- production-версия использует PostgreSQL, а опубликованная ранее версия GitHub Pages остаётся автономным демо;
- все деньги в Prisma хранятся как `Decimal(12,2)`;
- документы представлены метаданными и URL, фактическое хранилище не навязывается;
- QR формируется в банковском формате `ST00012`; сумма передаётся в копейках;
- до заполнения настоящих реквизитов формируется только безопасный демо-код;
- одна валюта (RUB), одна компания, авансы без привязки к начислению разрешены;
- удаление заменено явными статусами `archived` / `isActive`.

## Осознанно не реализовано

Нет публичного бронирования, клиентского кабинета, CRM-воронки, уведомлений, календаря, онлайн-кассы, 1С, SMS automation, генерации PDF/DOCX, электронной подписи, multi-tenant архитектуры, сложного BI, импорта/экспорта и audit log. Автоматическая отправка email и обработка входящих чеков требуют подключения серверного почтового backend.
